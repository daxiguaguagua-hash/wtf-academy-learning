## Git 结构笔记

### 为什么会有提交警告

原因：

```mermaid
flowchart LR

B --> C[lib/forge-std 是 submodule]
```

### 3. VS Code 导入警告排查

现象：

| 文件 | 提示 |
|---|---|
| `script/Counter.s.sol` | `Expected import path` |

原因：

| 原因 | 说明 |
|---|---|
| Foundry 使用 remapping | `forge-std/` 实际映射到 `lib/forge-std/src/` |
| VS Code 插件未立即识别 remapping | 所以对 `forge-std/Script.sol` 误报 |

排查结论：

```mermaid
flowchart LR
A[脚本代码本身正常] --> B[Foundry remapping 存在]
B --> C[编辑器未识别 remapping]
C --> D[出现 Expected import path]
```

解决方法：

```bash
cd hello-web3
forge remappings > remappings.txt
```

结果：

| 项目 | 结果 |
|---|---|
| `remappings.txt` | 已生成 |
| VS Code 导入警告 | 已消失 |

这一步学到的知识：
- `forge-std/Script.sol` 是 Foundry 的正常导入方式。
- `remappings.txt` 用来帮助编辑器理解依赖别名。
- 有些 Solidity 插件即使是最新版本，也可能需要项目显式提供 remapping 文件。

补充说明：
- 当前打开的 VS Code 工作区根目录是外层学习目录，不是 `hello-web3` 项目根目录。
- 当 Foundry 项目位于工作区子目录时，编辑器有时不会稳定识别 `foundry.toml` 和 `remappings.txt`。
- 这种情况下，可以在工作区补 `.vscode/settings.json`，手动告诉扩展依赖目录和 remapping。

当前工作区补充配置：

```json
{
  "solidity.packageDefaultDependenciesContractsDirectory": "src",
  "solidity.packageDefaultDependenciesDirectory": [
    "hello-web3/lib",
    "hello-web3/node_modules"
  ],
  "solidity.remappings": [
    "forge-std/=hello-web3/lib/forge-std/src/"
  ]
}
```

## ABI 与 cast 命令补充笔记

### ABI 是什么

| 缩写 | 全名 |
|---|---|
| `ABI` | `Application Binary Interface` |

一句话理解：
- ABI 是外部程序如何调用合约、以及如何解码返回结果的接口说明书。

### cast 里的函数签名模板

```text
函数名(参数类型1,参数类型2,...)(返回类型1,返回类型2,...)
```

例子：

| 写法 | 含义 |
|---|---|
| `"returnOne()(uint256)"` | 无参数，返回一个 `uint256` |
| `"returnMany()(uint256,bool,uint256)"` | 无参数，返回三个值 |
| `"numbers(uint256)(uint256)"` | 传一个 `uint256`，返回一个 `uint256` |
| `"sum(uint256[])(uint256)"` | 传一个 `uint256[]`，返回一个 `uint256` |

### 为什么不能只写函数名

原因：
- `cast` 需要知道参数类型，才能按 ABI 编码。
- `cast` 需要知道返回类型，才能按 ABI 解码。
- Solidity 支持函数重载，同名函数可能不止一个。

### cast call 的流程

```mermaid
flowchart LR
A[写函数签名] --> B[cast 按 ABI 编码]
B --> C[发给合约]
C --> D[合约执行]
D --> E[返回字节数据]
E --> F[cast 按 ABI 解码]
```

一句话结论：
- `cast` 里的类型声明，本质上是在使用 ABI。

### 默认值说明

| value 类型 | 默认值 |
|---|---|
| `uint256` | `0` |
| `bool` | `false` |
| `address` | `0x0000000000000000000000000000000000000000` |

说明：
- `mapping` 没有“报 key 不存在”这一说，而是直接返回 value 类型的默认值。
- 所以默认值有没有意义，取决于你的业务语义，不是 `mapping` 自己替你区分“真的写了默认值”还是“从没写过”。
- 如果业务上必须区分“没写过”和“写过但值刚好等于默认值”，通常要额外加一个标记，比如 `mapping(address => bool) hasValue`。

### 易错点

| 易错说法 | 正确说法 |
|---|---|
| `getBalance()` 比自动 getter 更安全，因为能防止别人读取 | 链上公开状态本来就是可读的；当前它们主要差别是可扩展性和可读性 |

结论：
- `mapping` 很适合余额、权限、授权、拥有者关系这类按 key 查询的场景。
- `mapping` 不能直接遍历，也不能直接知道“有多少项”。

### 本章关键点

| 关键词 | 含义 |
|---|---|

| `immutable` | 构造阶段赋值一次，之后不能改 |
| `msg.sender` | 当前调用者；在构造函数里通常就是部署者 |

### 代码理解

| 代码 | 作用 |
|---|---|
| `uint256 public constant MY_NUMBER = 123;` | 编译时固定常量 |
| `address public immutable owner;` | 部署时确定拥有者 |
| `owner = msg.sender;` | 把部署者地址写进 `owner` |

### 命令行验证结果

| 调用 | 结果 |
|---|---|
| `MY_NUMBER()` | `123` |

| 易错说法 | 正确说法 |
|---|---|
| `public` 和 `constant` 是一个维度的东西 | `public` 是可见性；`constant` / `immutable` 是可变性 |
| `immutable` 是部署后还能改一次 | `immutable` 只能在构造阶段赋值，部署完成后就不能改 |

### 补充理解

| 对比 | 更像什么 |
|---|---|
| `constant` | 写死在代码里的常量 |
| `immutable` | 部署时注入的只读配置 |
| 普通状态变量 | 部署后还能被函数修改的链上状态 |

结论：
- `constant` 和 `immutable` 都是“以后不能改”的变量。
- 区别在于：`constant` 在写代码时确定，`immutable` 在部署时确定。

| 题目 | 正确说法 |

| 写法 | 含义 |

| 再读 `number()` | `7` |
| 非 owner 调 `setNumber(99)` | 回退：`Not the owner` |

| 易错说法 | 正确说法 |
|---|---|
| `constructor` 和普通函数差不多，只是名字特殊 | `constructor` 只在部署时自动执行一次 |

### 事件日志理解

| 日志字段 | 含义 |
|---|---|
| `address` | 发出这条日志的合约地址 |
| `topics[0]` | 事件签名哈希 |
| `topics[1]` | `indexed caller` |
| `data` | `newCount` |

### 易错点

| 易错说法 | 正确说法 |
|---|---|
| `forge create` 跑了就一定已经部署成功 | 没加 `--broadcast` 时只是 dry run |
| `topics[0]` 是合约地址 | 合约地址在 `address` 字段；`topics[0]` 是事件签名哈希 |
| 前端监听到事件后就不用再读状态 | 常见做法是先读一次状态，再监听事件，事件触发后再同步状态 |
| `Out of gas: ... allowance: 0` 一定是合约逻辑错了 | 这类常见于账户没钱、私钥不对，甚至还没进合约执行阶段 |
| 切到本地网络就一定能发交易 | 还不够；当前钱包账户里还需要有本地链测试 ETH |
| 读合约也必须连接钱包 | 不一定；只读调用可以直接走 RPC |
| 前端点完按钮就算交易完成 | 还要等待 receipt，确认真正上链 |

### 当前章节结论

- 最小 DApp 合约至少要有：可读状态、可写函数、事件、自定义错误。
- `call` 解决“现在链上是多少”，`event` 解决“链上刚刚发生了什么变化”。
- 前端后续接入时，会同时用 ABI 来调用函数、解码错误、解码事件。
- 前端读链和前端写链是两套能力：读链靠 `publicClient`，写链靠钱包签名 + `walletClient`。
- 本地 DApp 调试时，网络切换、账户余额、交易确认，这三件事缺一不可。

## DApp 最小实战 第 1 节补充：本地链重置与恢复

### 现象

| 现象 | 实际表现 |
|---|---|
| MetaMask 里的本地测试 ETH 没了 | 之前到账的余额消失 |
| 前端 `count` / `owner` 读不到 | 页面显示 `-` |
| 状态消息报错 | `读取失败，请检查本地链和合约地址` |

### 根因

| 检查项 | 结果 | 说明 |
|---|---|---|
| `cast block-number` | `0` | 当前是刚启动的新 `anvil` 链 |
| `cast code <旧合约地址>` | `0x` | 旧地址上已经没有合约代码 |
| `cast balance <anvil默认账户>` | `10000000000000000000000` | 默认测试账户回到初始余额 |

### 为什么会这样

| 结论 | 说明 |
|---|---|
| `anvil` 默认是临时本地链 | 不是永久保存状态的正式网络 |
| 关闭或重启 `anvil` 进程 | 会得到一条新的本地链实例 |
| 问题不是“旧数据被篡改” | 而是“已经不在旧链上了” |

### 恢复命令

```bash
cast block-number --rpc-url http://127.0.0.1:8545
cast code 0x5FbDB2315678afecb367f032d93F642f64180aa3 --rpc-url http://127.0.0.1:8545
cast balance 0xf39Fd6e51aad88F6F4ce6aB8827279cffFb92266 --rpc-url http://127.0.0.1:8545
```

```bash
forge create src/MinimalDapp.sol:MinimalDapp \
  --rpc-url http://127.0.0.1:8545 \
  --private-key 0xac0974bec39a17e36ba4a6b4d238ff944bacb478cbed5efcae784d7bf4f2ff80 \
  --broadcast
```

```bash
cast send 0x68269ebf49B17232A806E4CAf126b340064D24ad \
  --value 10ether \
  --rpc-url http://127.0.0.1:8545 \
  --private-key 0xac0974bec39a17e36ba4a6b4d238ff944bacb478cbed5efcae784d7bf4f2ff80
```

### 恢复结果

| 项目 | 结果 |
|---|---|
| 重新部署后的合约地址 | 仍然是 `0x5FbDB2315678afecb367f032d93F642f64180aa3` |
| 给 MetaMask 账户补 ETH | 成功，当前恢复到 `10 ETH` |
| 前端重新读取 `count` | `0` |
| 前端重新读取 `owner` | `0xf39Fd6e51aad88F6F4ce6aB8827279cffFb92266` |

### 易错点补充

| 易错说法 | 正确说法 |
|---|---|
| 链上写入后就永远不会丢 | 要先问“是不是同一条链” |
| MetaMask 里网络名字没变，链状态就不会变 | 网络配置名不等于链状态持久化 |
| 本地链重置说明代码坏了 | 更常见是 `anvil` 被重启，状态回到新链初始值 |

### 补充结论

- 本地 `anvil` 更像临时测试链，不像主网/测试网那样天然持久。
- 遇到“余额消失 + 合约读不到”时，要先检查是不是已经切换到了新的本地链实例。
- 恢复顺序通常是：确认链状态 -> 重新部署 -> 补测试 ETH -> 刷新前端。

## DApp 最小实战 第 1 节补充：`setCount(5)` 为什么失败

### 当前结论

| 问题 | 结论 |
|---|---|
| 为什么 `increment()` 能成功 | 因为它没有 owner 权限限制 |
| 为什么 `setCount(5)` 会失败 | 因为它要求 `msg.sender == owner` |
| 当前 MetaMask 账户能不能调用 `setCount(5)` | 不能，因为不是 owner |
| 这是前端 bug 吗 | 不是，这是合约权限规则导致的预期失败 |

### 关键代码理解

| 代码 | 含义 |
|---|---|
| `owner` | 部署合约时记录下来的管理员地址 |
| `msg.sender` | 当前发起这笔交易的钱包地址 |
| `if (msg.sender != owner) { revert NotOwner(); }` | 不是 owner 就拒绝执行 |

### 本节收口

- `increment()` 当前承担“成功写交易”的教学角色。
- `setCount(5)` 当前承担“权限失败 + 错误提示”的教学角色。
- 前端不能把“钱包点了 Confirm”和“合约执行成功”当成一回事。

### 这一步只学一个最小知识点

| 层级 | 作用 |
|---|---|
| 发交易前提示 | 先告诉用户“这次大概率会成功还是会失败” |
| 链上错误解码 | 真失败后，把 `NotOwner()`、`CountTooSmall(...)` 翻译成人话 |

最小流程：

```mermaid
flowchart LR

B --> C[请求钱包签名]
```

### 当前前端行为

| 场景 | 页面提示 |
|---|---|
| 未连接钱包 | 先提示连接钱包 |
| 当前账户不是 owner | 预期提示：预计抛出 `NotOwner()` |
| 当前账户是 owner，但 `count >= 5` | 预期提示：预计抛出 `CountTooSmall(...)` |
| 钱包里点取消 | 提示“你在钱包里取消了这笔交易”，且不会进入链上执行 |
| 合约成功执行 | 等 receipt 后提示交易确认成功 |

### 关键代码理解

| 代码/概念 | 作用 |
|---|---|
| `getSetCountExpectation()` | 先根据当前账户和 `count`，判断这笔交易大概率会走哪条分支 |
| `getReadableErrorMessage(error)` | 把 viem 捕获到的底层错误翻译成中文 |
| `error.walk(...)` | 沿错误链往里找真正的 revert 原因 |
| `error.shortMessage` | 兜底显示底层库给出的简短错误信息 |
| `ContractFunctionRevertedError.data.errorName` | `viem` 解码自定义错误后，真正放错误名的位置 |

### 命令

```bash
cd hello-web3/frontend
npm run build
```

### 易错点

| 易错说法 | 正确说法 |
|---|---|
| 前端提前知道会失败，就不要真的发交易了 | 教学阶段可以保留发送流程，用来观察真实 revert 如何被前端翻译 |
| 钱包弹窗出现就说明已经成功 | 钱包确认只是“准备发送”，不是“链上已成功执行” |
| 错误提示只能看 `error.message` | 更好的做法是根据 ABI 和错误类型做结构化翻译 |
| 现在先学错误提示，也必须顺便把事件监听做完 | 不需要，本轮可以先把错误主线单独收口 |
| `NotOwner()` 一定会出现在 `error.cause.name` | 不一定；在当前 `viem` 版本里，更稳的是读 `ContractFunctionRevertedError.data.errorName` |

### 当前章节结论

- 本轮先把“失败交易怎么解释给用户听”单独学清楚。
- 前端错误处理至少要区分三类：用户取消、权限失败、参数失败。
- `setCount(5)` 现在已经能同时承担“预判失败”和“翻译链上错误”这两个教学角色。
- 浏览器控制台里会看到 `UserRejectedRequestError` 错误链，这说明失败点在钱包签名阶段，不是在合约内部。
- 如果自定义错误已经写进 ABI，但页面仍没识别，优先检查是不是取错了 `viem` 错误层级。

## DApp 最小实战 第 1 节补充：事件监听最小版

### 当前实操结果

| 项目 | 状态 |
|---|---|
| `watchContractEvent` 接回页面 | 已完成 |
| 监听事件名 | `CountUpdated` |

| `vite build` | 已通过 |

### 这一步只学一个最小知识点

| 方式 | 什么时候刷新页面 |
|---|---|
| 手动重新读取 | 当前页面自己发交易成功后 |
| 事件监听刷新 | 只要链上发出目标事件就刷新 |

流程对比：

```mermaid
flowchart LR
A[交易成功] --> B[waitForTransactionReceipt]
B --> C[loadContractData]
```

```mermaid
flowchart LR
A[链上 emit CountUpdated] --> B[watchContractEvent 收到日志]
B --> C[loadContractData]
```

### 关键代码理解

| 代码 | 作用 |
|---|---|
| `publicClient.watchContractEvent(...)` | 开始监听指定合约事件 |
| `eventName: "CountUpdated"` | 只关心 `CountUpdated` 这类日志 |
| `onLogs(logs) { ... }` | 一旦收到事件，执行回调 |
| `unwatchContractEvent()` | 页面卸载时停止监听，避免重复订阅 |

### 易错点

| 易错说法 | 正确说法 |
|---|---|
| 页面数据变了就一定是事件监听生效 | 也可能只是交易成功后手动调用了 `loadContractData()` |
| 监听事件后就完全不用主动读链 | 常见做法仍然是“先读一次初始状态，再监听后续变化” |
| 事件里已经有 `newCount`，就一定不用重新读链 | 教学阶段继续重新读链更直观，也更稳 |

### 当前章节结论

- 你现在已经同时拥有两种刷新方式：交易成功后手动刷新、收到事件后自动刷新。
- 事件监听的价值在于：不要求一定是“当前页面自己发出的交易”。
- 这一节到这里，读、写、确认、错误、事件这 5 条最小主线已经基本串起来了。
