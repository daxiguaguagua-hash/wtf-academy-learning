### 为什么会有提交警告

原因：

```mermaid
flowchart LR
```

### 3. VS Code 导入警告排查

现象：

| 文件 | 提示 |
|---|---|
| `script/Counter.s.sol` | `Expected import path` |

原因：

| 原因 | 说明 |
|---|---|
| Foundry 使用 remapping | `forge-std/` 实际映射到 `lib/forge-std/src/` |
| VS Code 插件未立即识别 remapping | 所以对 `forge-std/Script.sol` 误报 |

排查结论：

```mermaid
flowchart LR
A[脚本代码本身正常] --> B[Foundry remapping 存在]
B --> C[编辑器未识别 remapping]
C --> D[出现 Expected import path]
```

解决方法：

```bash
cd hello-web3
forge remappings > remappings.txt
```

结果：

| 项目 | 结果 |
|---|---|
| `remappings.txt` | 已生成 |
| VS Code 导入警告 | 已消失 |

这一步学到的知识：
- `forge-std/Script.sol` 是 Foundry 的正常导入方式。
- `remappings.txt` 用来帮助编辑器理解依赖别名。
- 有些 Solidity 插件即使是最新版本，也可能需要项目显式提供 remapping 文件。

补充说明：
- 当前打开的 VS Code 工作区根目录是外层学习目录，不是 `hello-web3` 项目根目录。
- 当 Foundry 项目位于工作区子目录时，编辑器有时不会稳定识别 `foundry.toml` 和 `remappings.txt`。
- 这种情况下，可以在工作区补 `.vscode/settings.json`，手动告诉扩展依赖目录和 remapping。

当前工作区补充配置：

```json
{
  "solidity.packageDefaultDependenciesContractsDirectory": "src",
  "solidity.packageDefaultDependenciesDirectory": [
    "hello-web3/lib",
    "hello-web3/node_modules"
  ],
  "solidity.remappings": [
    "forge-std/=hello-web3/lib/forge-std/src/"
  ]
}
```

## ABI 与 cast 命令补充笔记

### ABI 是什么

| 缩写 | 全名 |
|---|---|
| `ABI` | `Application Binary Interface` |

一句话理解：
- ABI 是外部程序如何调用合约、以及如何解码返回结果的接口说明书。

### cast 里的函数签名模板

```text
函数名(参数类型1,参数类型2,...)(返回类型1,返回类型2,...)
```

例子：

| 写法 | 含义 |
|---|---|
| `"returnOne()(uint256)"` | 无参数，返回一个 `uint256` |
| `"returnMany()(uint256,bool,uint256)"` | 无参数，返回三个值 |
| `"numbers(uint256)(uint256)"` | 传一个 `uint256`，返回一个 `uint256` |
| `"sum(uint256[])(uint256)"` | 传一个 `uint256[]`，返回一个 `uint256` |

### 为什么不能只写函数名

原因：
- `cast` 需要知道参数类型，才能按 ABI 编码。
- `cast` 需要知道返回类型，才能按 ABI 解码。
- Solidity 支持函数重载，同名函数可能不止一个。

### cast call 的流程

```mermaid
flowchart LR
A[写函数签名] --> B[cast 按 ABI 编码]
B --> C[发给合约]
C --> D[合约执行]
D --> E[返回字节数据]
E --> F[cast 按 ABI 解码]
```

一句话结论：
- `cast` 里的类型声明，本质上是在使用 ABI。

### 默认值说明

| value 类型 | 默认值 |
|---|---|
| `uint256` | `0` |
| `bool` | `false` |
| `address` | `0x0000000000000000000000000000000000000000` |

说明：
- `mapping` 没有“报 key 不存在”这一说，而是直接返回 value 类型的默认值。
- 所以默认值有没有意义，取决于你的业务语义，不是 `mapping` 自己替你区分“真的写了默认值”还是“从没写过”。
- 如果业务上必须区分“没写过”和“写过但值刚好等于默认值”，通常要额外加一个标记，比如 `mapping(address => bool) hasValue`。

### 易错点

| 易错说法 | 正确说法 |
|---|---|
| `getBalance()` 比自动 getter 更安全，因为能防止别人读取 | 链上公开状态本来就是可读的；当前它们主要差别是可扩展性和可读性 |

结论：
- `mapping` 很适合余额、权限、授权、拥有者关系这类按 key 查询的场景。
- `mapping` 不能直接遍历，也不能直接知道“有多少项”。

### 本章关键点

| 关键词 | 含义 |

| `immutable` | 构造阶段赋值一次，之后不能改 |
| `msg.sender` | 当前调用者；在构造函数里通常就是部署者 |

### 代码理解

| 代码 | 作用 |
|---|---|
| `uint256 public constant MY_NUMBER = 123;` | 编译时固定常量 |
| `address public immutable owner;` | 部署时确定拥有者 |

### 命令行验证结果

| 调用 | 结果 |
|---|---|
| `MY_NUMBER()` | `123` |

| 易错说法 | 正确说法 |
|---|---|
| `public` 和 `constant` 是一个维度的东西 | `public` 是可见性；`constant` / `immutable` 是可变性 |

### 补充理解

| 对比 | 更像什么 |
|---|---|
| `constant` | 写死在代码里的常量 |
| `immutable` | 部署时注入的只读配置 |
| 普通状态变量 | 部署后还能被函数修改的链上状态 |

结论：
- `constant` 和 `immutable` 都是“以后不能改”的变量。
- 区别在于：`constant` 在写代码时确定，`immutable` 在部署时确定。

| 易错说法 | 正确说法 |
|---|---|

### 事件日志理解

| 日志字段 | 含义 |
|---|---|
| `address` | 发出这条日志的合约地址 |
| `topics[0]` | 事件签名哈希 |
| `topics[1]` | `indexed caller` |
| `data` | `newCount` |

### 易错点

| 易错说法 | 正确说法 |
|---|---|
| `forge create` 跑了就一定已经部署成功 | 没加 `--broadcast` 时只是 dry run |
| `topics[0]` 是合约地址 | 合约地址在 `address` 字段；`topics[0]` 是事件签名哈希 |
| 前端监听到事件后就不用再读状态 | 常见做法是先读一次状态，再监听事件，事件触发后再同步状态 |
| `Out of gas: ... allowance: 0` 一定是合约逻辑错了 | 这类常见于账户没钱、私钥不对，甚至还没进合约执行阶段 |
| 切到本地网络就一定能发交易 | 还不够；当前钱包账户里还需要有本地链测试 ETH |
| 读合约也必须连接钱包 | 不一定；只读调用可以直接走 RPC |
| 前端点完按钮就算交易完成 | 还要等待 receipt，确认真正上链 |

### 当前章节结论

- 最小 DApp 合约至少要有：可读状态、可写函数、事件、自定义错误。
- `call` 解决“现在链上是多少”，`event` 解决“链上刚刚发生了什么变化”。
- 前端后续接入时，会同时用 ABI 来调用函数、解码错误、解码事件。
- 前端读链和前端写链是两套能力：读链靠 `publicClient`，写链靠钱包签名 + `walletClient`。
- 本地 DApp 调试时，网络切换、账户余额、交易确认，这三件事缺一不可。

## DApp 最小实战 第 1 节补充：`setCount(5)` 为什么失败

### 当前结论

| 问题 | 结论 |
|---|---|
| 为什么 `increment()` 能成功 | 因为它没有 owner 权限限制 |

| `if (msg.sender != owner) { revert NotOwner(); }` | 不是 owner 就拒绝执行 |

### 本节收口

- `increment()` 当前承担“成功写交易”的教学角色。

- 事件监听的价值在于：不要求一定是“当前页面自己发出的交易”。
- 这一节到这里，读、写、确认、错误、事件这 5 条最小主线已经基本串起来了。
- 如果改完监听代码后没看到日志，先重启前端页面/浏览器，再判断是不是监听本身有问题。

| 只要大概是同一组数据，hash 应该差不多 | 不对，输入改一点，hash 就会大变 |
| `broadcast` 里的旧地址一定等于当前链实例上的合约地址 | 不一定，切换或重启 `anvil` 后要重新对齐当前链状态 |

### 当前章节结论

- `keccak256(abi.encode(...))` 是 Solidity 里最常见的哈希写法。
- 相同输入得到相同哈希，不同输入通常会得到完全不同的哈希。
- 这章最重要的是理解“先编码，再哈希”的工作流。

| 场景 | 结果 |
|---|---|
| `trySetNumber(target, 7)` | `lastResult = 7`，`lastMessage = success`，`lastSuccess = true`，`target.number = 7` |

### 易错点

| 易错说法 | 正确说法 |
|---|---|
| `try ... returns (...)` 写的是当前函数自己的返回值 | 不是，写的是被 `try` 的那次外部调用成功时的返回值 |
| 目标合约报错了，外层合约一定整体崩掉 | 不一定，`try catch` 可以接住 |
| 失败分支执行后，目标合约状态也会变成失败时传入的值 | 不会，失败那次不会写入目标状态 |

### 当前章节结论

- `try catch` 是跨合约失败处理的更高层写法。
- 成功时可以直接接住返回值，失败时可以走 `catch` 分支自己兜底。
- 这章最重要的是理解“目标合约失败了，但调用方还能继续定义自己的处理逻辑”。
